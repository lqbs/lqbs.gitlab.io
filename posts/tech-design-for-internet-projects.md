技术设计的核心目标，是把产品需求转成一套「研发可落地、测试可验证、上线可运维、后期可扩展」的实施方案。

这一步是开发阶段最关键的前置工作——方案做扎实了，开发基本是顺水推舟；方案糊弄过去，所有问题都会在联调、测试、上线阶段集中爆发。

下面按 18 个步骤展开，每一步都列出「关注点、常见做法、输出物」三块内容。

## 1. 需求理解与技术拆解

技术设计不是一上来就画架构图，而是先理解需求。

需要确认：

| 关注点 | 说明 |
| --- | --- |
| 业务目标 | 这个系统解决什么问题 |
| 核心流程 | 用户、后台、第三方系统之间怎么流转 |
| 功能范围 | 哪些做，哪些不做 |
| 用户规模 | 日活、并发、访问峰值预估 |
| 数据规模 | 数据量、增长速度、存储周期 |
| 性能要求 | 页面响应、接口响应、任务处理时效 |
| 安全要求 | 登录、权限、数据加密、防攻击 |
| 合规要求 | 隐私、日志、数据留存、支付规范等 |
| 上线时间 | 是否需要分阶段交付 |

输出物：

- 技术需求分析说明
- 技术风险清单
- 待确认问题列表

这一阶段把需求含糊的地方都问清楚，比后面返工成本低得多。

## 2. 系统边界设计

明确系统要做什么、不做什么，以及和哪些外部系统交互。

需要设计：

| 类型 | 示例 |
| --- | --- |
| 用户端 | Web、H5、小程序、App |
| 管理端 | 后台管理系统、运营平台 |
| 服务端 | 用户服务、订单服务、内容服务、支付服务 |
| 第三方系统 | 微信登录、支付宝、短信服务、地图服务、对象存储 |
| 基础设施 | 数据库、缓存、消息队列、日志、监控 |

重点要明确：

- 哪些功能在前端完成
- 哪些逻辑在后端完成
- 哪些能力依赖第三方
- 哪些数据由本系统维护
- 哪些数据从外部系统同步
- 系统之间通过什么方式通信

输出物：

- 系统上下文图
- 系统边界说明
- 外部依赖清单

边界没划清，团队之间就会出现「这事该谁做」的反复扯皮。

## 3. 总体架构设计

技术设计的核心部分。

### 单体架构

适合：项目早期、团队较小、功能不复杂、上线周期紧。

开发快、部署简单。缺点是后期模块耦合容易变高。

### 前后端分离架构

适合大多数互联网项目。典型结构：

```
用户端 / 管理端
      ↓
前端应用
      ↓
API 网关 / 后端接口
      ↓
业务服务
      ↓
数据库 / 缓存 / 文件存储 / 消息队列
```

### 微服务架构

适合：业务复杂、团队规模较大、系统访问量高、服务需要独立扩展、业务边界清晰。

常见拆分：

- 用户服务
- 商品服务
- 订单服务
- 支付服务
- 内容服务
- 消息服务
- 运营服务

注意：不是所有项目都适合微服务。早期项目盲目微服务，反而会增加复杂度。

输出物：

- 系统架构图
- 技术架构说明
- 服务拆分方案
- 部署架构草案

## 4. 技术选型

根据团队能力、项目周期、性能要求和维护成本选型。

### 前端

| 方向 | 可选方案 |
| --- | --- |
| Web 前端 | Vue、React、Angular |
| 移动端 | Flutter、React Native、原生 App |
| 小程序 | 微信小程序、uni-app、Taro |
| 管理后台 | Vue Admin、React Admin、自研后台 |
| 构建工具 | Vite、Webpack |
| 状态管理 | Pinia、Redux、Zustand |
| UI 组件库 | Ant Design、Element Plus、Naive UI |

### 后端

| 方向 | 可选方案 |
| --- | --- |
| 开发语言 | Java、Go、Node.js、Python、PHP |
| 框架 | Spring Boot、Gin、NestJS、Django、Laravel |
| API 风格 | RESTful、GraphQL、gRPC |
| 认证 | JWT、Session、OAuth2、SSO |
| ORM | MyBatis、JPA、Prisma、SQLAlchemy |

### 数据与中间件

| 类型 | 常见方案 |
| --- | --- |
| 关系数据库 | MySQL、PostgreSQL |
| 缓存 | Redis |
| 搜索 | Elasticsearch、OpenSearch |
| 消息队列 | RabbitMQ、Kafka、RocketMQ |
| 对象存储 | S3、OSS、COS、MinIO |
| 日志 | ELK、Loki |
| 监控 | Prometheus、Grafana、云监控 |

技术选型要说明：

- 为什么选它
- 不选其他方案的原因
- 团队是否熟悉
- 是否满足性能要求
- 后期维护成本如何
- 是否存在授权或成本问题

输出物：

- 技术选型说明
- 技术风险评估
- 第三方服务选型表

## 5. 模块划分设计

把系统拆成可开发、可测试、可维护的模块。

以电商系统为例：

| 模块 | 主要功能 |
| --- | --- |
| 用户模块 | 注册、登录、账号、安全设置 |
| 商品模块 | 商品管理、分类、库存、上下架 |
| 购物车模块 | 加购、修改数量、删除 |
| 订单模块 | 下单、取消、支付状态、售后 |
| 支付模块 | 支付发起、回调、退款 |
| 营销模块 | 优惠券、满减、活动 |
| 内容模块 | Banner、文章、公告 |
| 消息模块 | 短信、站内信、邮件、推送 |
| 后台模块 | 权限、角色、菜单、运营管理 |

每个模块要明确：

- 模块职责
- 输入输出
- 依赖关系
- 核心接口
- 核心数据表
- 异常处理
- 权限控制

输出物：

- 模块设计说明
- 服务职责说明
- 模块依赖图

## 6. 数据库设计

数据库设计直接影响系统性能和后期扩展。

### 6.1 概念模型设计

先识别核心实体：

- 用户、角色、商品、订单、支付单、优惠券、文章、文件、操作日志

定义实体关系：

```
用户 1 - N 订单
订单 1 - N 订单明细
订单 1 - 1 支付单
用户 N - N 角色
角色 N - N 权限
```

### 6.2 逻辑模型设计

```sql
CREATE TABLE user (
  id BIGINT PRIMARY KEY,
  username VARCHAR(64) NOT NULL,
  phone VARCHAR(20),
  password_hash VARCHAR(255) NOT NULL,
  status TINYINT NOT NULL DEFAULT 1,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL
);
```

### 6.3 字段设计

每个字段要明确：

- 字段名、数据类型、是否必填、默认值
- 唯一约束、字段含义
- 是否可为空、是否需要脱敏、是否参与查询

### 6.4 索引设计

重点关注：

- 高频查询字段
- 唯一性字段
- 排序字段
- 联表字段
- 状态字段和时间字段组合查询

```sql
CREATE INDEX idx_order_user_status_time 
ON orders(user_id, status, created_at);
```

### 6.5 数据一致性设计

需要考虑：

- 是否使用事务
- 是否存在分布式事务
- 是否允许最终一致性
- 支付回调如何防重复
- 库存扣减如何防超卖
- 订单状态如何流转

输出物：

- ER 图
- 数据库设计文档
- 建表 SQL
- 索引设计说明
- 数据字典

## 7. 接口设计

接口设计是前后端联调的依据。

每个接口至少要定义：

| 内容 | 示例 |
| --- | --- |
| 接口名称 | 用户登录 |
| 请求方式 | POST |
| URL | /api/v1/auth/login |
| 请求参数 | phone、password |
| 响应字段 | token、userInfo |
| 错误码 | 账号不存在、密码错误 |
| 权限要求 | 无需登录 |
| 幂等要求 | 是否允许重复提交 |
| 限流要求 | 每分钟最多多少次 |
| 日志要求 | 是否记录操作日志 |

### 请求示例

```http
POST /api/v1/auth/login
Content-Type: application/json

{
  "phone": "13800000000",
  "password": "123456"
}
```

### 响应示例

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "token": "xxx",
    "user": {
      "id": 10001,
      "nickname": "张三"
    }
  }
}
```

输出物：

- API 接口文档
- 错误码文档
- 接口 Mock 数据
- 联调说明

## 8. 权限与安全设计

互联网项目不能后补安全，技术设计阶段就要考虑。

### 8.1 认证设计

常见方式：

- Session + Cookie
- JWT
- OAuth2
- 单点登录 SSO
- 微信 / 支付宝 / Apple 登录

要明确：

- 登录态有效期
- Token 刷新机制
- 多端登录策略
- 退出登录机制
- 密码加密方式
- 验证码策略

### 8.2 授权设计

| 模型 | 说明 |
| --- | --- |
| RBAC | 用户 - 角色 - 权限 |
| ABAC | 基于属性授权 |
| ACL | 针对资源单独授权 |

大多数后台系统使用 RBAC：

```
用户 → 角色 → 权限 → 菜单 / 按钮 / 接口 / 数据范围
```

### 8.3 安全防护

需要考虑：

- SQL 注入、XSS、CSRF
- 暴力破解、接口刷量
- 文件上传风险
- 敏感数据脱敏
- HTTPS
- 数据库密码加密
- 日志中不能打印密码、Token、身份证号等敏感信息

输出物：

- 认证授权方案
- 权限模型设计
- 安全设计说明
- 敏感数据清单

## 9. 性能与高可用设计

这一部分决定系统上线后能不能扛住流量。

### 9.1 性能设计

| 场景 | 方案 |
| --- | --- |
| 首页访问慢 | CDN、缓存、接口聚合 |
| 热点数据查询 | Redis 缓存 |
| 大列表查询 | 分页、索引、异步加载 |
| 搜索慢 | Elasticsearch |
| 文件访问慢 | 对象存储 + CDN |
| 写入高峰 | 消息队列削峰 |
| 大任务慢 | 异步任务 |
| 统计报表慢 | 离线计算、预聚合 |

### 9.2 高可用设计

需要考虑：

- 服务多实例部署
- 负载均衡
- 数据库主从
- Redis 高可用
- 消息队列持久化
- 服务降级、熔断限流
- 自动重启、健康检查
- 容灾备份

### 9.3 容量评估

至少要估算：

- 日活用户数、峰值 QPS
- 数据库数据量、缓存容量
- 文件存储容量、带宽消耗
- 服务器配置

输出物：

- 性能设计方案
- 容量评估表
- 高可用设计说明
- 压测方案

## 10. 文件与存储设计

涉及图片、视频、附件、合同、头像等内容时，需要单独设计。

需要明确：

- 文件上传大小限制、文件类型限制
- 存储位置、文件命名规则
- 访问权限、防盗链策略
- 图片压缩、视频转码
- CDN 加速、临时文件清理
- 文件生命周期管理

常见方案：

```
前端上传 → 后端获取上传凭证 → 前端直传对象存储 → 后端保存文件记录
```

输出物：

- 文件存储方案
- 上传接口设计
- 文件访问权限说明

## 11. 日志、监控与告警设计

上线后能不能定位问题，取决于这一步。

### 日志设计

需要记录：

- 请求日志、错误日志、操作日志
- 登录日志、支付回调日志
- 第三方接口调用日志、定时任务执行日志

日志字段建议包括：

```
traceId
userId
requestId
接口路径
请求耗时
错误码
异常堆栈
客户端 IP
设备信息
时间戳
```

### 监控设计

需要监控：

- CPU / 内存 / 磁盘
- QPS、响应时间、错误率
- 数据库连接数、Redis 命中率
- 消息队列堆积
- 第三方接口失败率
- 定时任务状态

### 告警设计

常见告警条件：

- 接口错误率超过阈值
- 接口响应时间过长
- 服务不可用
- 磁盘空间不足
- 支付回调失败
- 消息队列严重堆积
- 数据库慢查询增加

输出物：

- 日志规范
- 监控指标清单
- 告警规则表

## 12. 部署架构设计

明确系统如何发布到服务器或云平台。

### 传统部署

```
Nginx
  ↓
前端静态资源
后端服务
  ↓
MySQL / Redis
```

### Docker 部署

```
Nginx
  ↓
Docker 容器
  ↓
MySQL / Redis / MQ
```

### Kubernetes 部署

适合复杂系统和多服务场景：

```
Ingress
  ↓
Service
  ↓
Pod
  ↓
ConfigMap / Secret / PVC
```

部署设计要明确：

- 环境划分：开发、测试、预发、生产
- 域名规划、SSL 证书
- 配置管理、镜像构建
- CI/CD 流程
- 数据库迁移、回滚机制
- 灰度发布、备份策略

输出物：

- 部署架构图
- 环境规划文档
- CI/CD 流程说明
- 上线与回滚方案

## 14. 异常处理与容错设计

系统一定会出错，所以要提前设计怎么失败。

常见异常：

- 网络超时、数据库连接失败
- Redis 不可用、第三方接口失败
- 支付回调重复、用户重复提交
- 库存不足、文件上传失败
- 消息消费失败

设计策略：

| 问题 | 处理方式 |
| --- | --- |
| 重复提交 | 幂等 Token、唯一索引 |
| 第三方超时 | 超时控制、重试、降级 |
| 支付重复回调 | 幂等处理、订单状态机 |
| 消息消费失败 | 重试队列、死信队列 |
| 数据写入失败 | 事务回滚 |
| 高并发请求 | 限流、排队、缓存 |
| 核心服务失败 | 熔断、降级、兜底提示 |

输出物：

- 异常处理方案
- 幂等设计说明
- 降级与熔断策略

- 日志规范
- 配置规范
- 数据库变更规范

```json
{

```

输出物：

- 开发规范文档

## 16. 技术评审

技术方案不是写完就结束，需要评审。

评审人员通常包括：

- 技术负责人
- 前端负责人、后端负责人
- 测试负责人
- 产品经理
- 运维 / DevOps
- 安全负责人

评审重点：

- 方案是否满足业务需求
- 架构是否过度设计
- 数据库设计是否合理
- 接口是否完整
- 性能是否能支撑目标
- 安全风险是否可控
- 上线方案是否可靠
- 排期是否现实
- 风险是否有预案

输出物：

- 技术评审记录
- 问题清单
- 方案修订版

---

## 17. 开发任务拆分与排期

技术方案通过后，要拆成可执行任务。

拆分维度：

- 按模块拆
- 按前后端拆
- 按接口拆
- 按页面拆
- 按数据库变更拆
- 按第三方接入拆
- 按测试联调拆

任务示例：

| 任务 | 负责人 | 工期 | 依赖 |
| --- | --- | --- | --- |
| 用户表设计 | 后端 | 0.5 天 | 无 |
| 登录接口开发 | 后端 | 1 天 | 用户表 |
| 登录页开发 | 前端 | 1 天 | UI 稿 |
| 登录接口联调 | 前后端 | 0.5 天 | 接口完成 |
| 登录测试 | 测试 | 0.5 天 | 联调完成 |

输出物：

- 研发任务清单
- 开发排期表
- 里程碑计划

## 18. 技术设计最终交付物

完整技术设计通常至少包括以下文档：

| 文档 | 作用 |
| --- | --- |
| 技术方案文档 | 总体说明如何实现 |
| 系统架构图 | 展示系统整体结构 |
| 模块设计文档 | 说明模块职责与边界 |
| 数据库设计文档 | 表结构、字段、索引、关系 |
| API 接口文档 | 前后端联调依据 |
| 部署架构文档 | 说明环境、服务器、发布方式 |
| 安全设计文档 | 认证、权限、数据安全 |
| 性能设计文档 | 缓存、限流、容量、压测 |
| 日志监控文档 | 上线后排障依据 |
| 第三方接入文档 | 外部服务调用方式 |
| 技术风险清单 | 记录风险与解决方案 |
| 开发规范文档 | 保证多人协作一致性 |

## 一句话总结

技术设计就是在正式开发前，把系统架构、技术选型、数据结构、接口、权限、安全、性能、部署、监控和风险全部想清楚，形成一套研发团队可以直接执行的技术落地方案。

做扎实了，开发是顺水推舟；糊弄过去，所有问题都会在联调和上线阶段集中爆发。
